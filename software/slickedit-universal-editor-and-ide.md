I learned to love this editor many years ago while I was working on a bigger project on the Solaris 
operating system. Back at this time, Sun [One] Studio was basically the only solution next to just 
using Vi(m) or (X)Emacs as your IDE and all I want to say is that it was a painful experience. Sun 
built fine hardware and an excellent OS, but their end-user software wasn't that great (OPEN LOOK, 
yes, I'm looking at you).
 
Back in those days, the number of cross-platform text editors was limited to the aforementioned 
(X)Emacs and various vi variants. Today, we have quite a number of modern cross-platform editors like 
Sublime Text, Atom or Visual Studio, but cross-platform nowadays means *„it works on the usual 
suspects”* - Windows, Linux and OS X. Slick Edit is still available for 7 different platforms 

## Why spend 300 bucks for an editor, there are plenty for free?

That's a good question. Why pay close to 300 bucks for an editor, when there are plenty of totally 
free or significantly cheaper alternatives? [Atom](https://atom.io), [Visual Studio 
Code](https://code.visualstudio.com/) or [Brackets](https://brackets.io) are totally free, while 
Sublime Text costs $80 per license and thus significantly less than Slick Edit Pro. They are all very 
good editors with lots of features, many interesting add-ons, support for many programming languages 
out of the box and some more with plugins. They are all multi platform, running on the „big 3” 
(Windows, Linux and OS X) 

Well, for one, because SlickEdit simply does more. It's not just an editor, it's basically a full IDE, 
but it's not bound to a specific language or build environment. It can use many build systems, it runs 
on 6 different operating systems and it supports a couple dozens of programming languages. There are 
arguably more specialized editors in some areas, but no other one I know of supports so many language 
with more than just syntax highlighting.

## What's cool with SlickEdit?

* Stability. It rarely, if ever, crashes. Nor does it leak memory or grow endlessly. It will use quite 
a lot of memory for symbol databases and various caches if instructed, but this is by design to help 
with performance and can be configured to be more resource-friendly on lower end systems.
 
* Flexibility. Almost everything is customizable and most visual and/or behavioral settings can be 
configured on a „per language” basis 

## And what's not so cool?

* It's complex. Seriously, it does have a learning curve. Not because of poor design, but because it 
simply can do so much. You'll probably spend many hours with configuring it down to the smallest
